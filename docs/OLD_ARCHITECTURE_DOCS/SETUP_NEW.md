## 🏗 System Architecture Overview

The system strictly operates on a **3-Folder Microservice Architecture**. 
1. **`frontend/` (The Student Storefront):** A React/Web application handled by Vercel. Accessed by students on their personal phones. It handles UI for file uploading and Cashfree Payments, routing those actions directly to the `backend/`.
2. **`backend/` (The Node.js Processor API):** A standalone Express Node.js server. It securely handles Cashfree Payment callbacks, processes PDFs via LibreOffice/pdf-lib, and uses the `firebase-admin` SDK to upload files to Firebase Storage. Finally, it generates a unique **4-Digit PIN** and writes the job to Firestore.
3. **`kiosk-tablet-frontend/` (The Dispenser UI):** A locked-down Next.js web interface deployed on the physical Lenovo Android tablet. It simply presents a giant numpad, queries Firebase for the user's **4-Digit PIN**, and then proxies the print job securely down to the local hardware.

*(There is also the physical local Raspberry Pi Print Server & Brother Printer layer receiving commands from the Kiosk).*

## Phase 1: Firebase Configuration (The Data Bridge)

Firebase sits in the middle. It securely stores the 4-digit PIN generated by the backend and lets the Kiosk Tablet UI read it.

### 1. Project Initialization
1. Go to [Firebase Console](https://console.firebase.google.com/) and create a new project.
2. Enable **Firestore**, **Firebase Storage**, and **Authentication**.
3. Generate a Web Client SDK configuration, AND a `serviceAccountKey.json` for the backend.

### 2. Required Firestore Schema
**`printJobs` (Collection)**
```json
{
  "pin": "1234", // The generated 4-digit PIN
  "documentUrl": "https://firebasestorage.app/...",
  "documentName": "assignment.pdf",
  "status": "pending", // pending -> printing -> completed
  "paymentStatus": "completed", // Verified by Cashfree
  "printSettings": { "copies": 1, "color": false, "duplex": false, "paperSize": "A4" },
  "createdAt": "timestamp"
}
```

## Phase 2: Wiring the 3 Folders

When connecting the provided source codes, ensure the following logic is strictly enforced:

### 1. `frontend/` Logic
* **Upload:** Users upload PDFs and choose page settings. 
* **Routing:** The checkout process triggers an API call (e.g., `POST /api/checkout`) to the `backend/`.
* **Return State:** Once the backend verifies the Cashfree payment, the frontend must receive and boldly display the `"1234"` **4-Digit PIN** on the screen.

### 2. `backend/` Logic
* **Payments:** Handle `cashfree-pg` session generation and webhook callbacks.
* **Storage Execution:** On successful payment, upload the PDF buffer directly to Firebase Storage using `firebase-admin`.
* **Database Execution:** Push the new job payload containing exactly the key `"pin"` into the `printJobs` Firestore collection.

### 3. `kiosk-tablet-frontend/` Logic
* **Validation:** Restrict the UI number-pad state so it exactly requires `length === 4` digits.
* **Firebase Query:** The Kiosk queries Firestore natively: `collection('printJobs').where("pin", "==", inputPin)`.
* **Hardware Trigger:** On success, fire an internal API route that securely `POST`s the PDF payload down to the Raspberry Pi over the local network via `NEXT_PUBLIC_FASTAPI_PRINT_URL`.

## Phase 3: Hardware Print Server (Raspberry Pi & CUPS)

For remote, silent printing, the system relies on a local Raspberry Pi. 

1. Connect the Pi to the exact same WiFi network as your physical Brother printer.
2. Install necessary print drivers and document converters via SSH:
   ```bash
   sudo apt update
   sudo apt install cups libcups2-dev pdftk qpdf libreoffice
   pip install fastapi uvicorn python-cups python-multipart
   ```
3. Start the Python FastAPI server that listens for Next.js webhooks. Ensure `NEXT_PUBLIC_FASTAPI_PRINT_URL` in your Vercel Kiosk points to this local IP address (e.g., `http://192.168.1.100:8000/print`).
4. Ensure the Brother Printer IP is registered in the Pi's CUPS administration page at `http://localhost:631`.

## Phase 4: The Kiosk Tablet Physical Setup (Lenovo Tab M9)

1. Install **Fully Kiosk Browser** from the Google Play Store onto the Lenovo tablet.
2. Open Fully Kiosk Browser settings and set the **Start URL** to your Kiosk Vercel URL.
3. Enable Kiosk Mode, hide status/nav bars, set sleep to **Never**, and lock with an admin PIN.

## 🎯 The Complete End-to-End Workflow Validation

1. **Upload & Pay:** A user uploads a PDF from their phone via `frontend/`. It hits the `backend/` API, talks to Cashfree, and the user pays.
2. **Token Generation:** The `backend/` creates `{ "pin": "1234", "status": "pending" }` in Firestore, and uploads the PDF. The user receives "1234" on their phone screen.
3. **Kiosk Entry:** The user walks up to the Lenovo tablet and taps "1234" on the Fully Kiosk Browser UI (`kiosk-tablet-frontend/`).
4. **Hardware Execution:** The Kiosk triggers the print API. The local Raspberry Pi fetches the file and orders the Brother printer to drop the paper. The Firestore status updates to `"completed"`.

## 🛠 AI Expert Code Generation Imperatives

**AI INSTRUCTION:** When instructed to build and integrate the logic files for this system, you MUST generate the code using precisely these rules and structures to guarantee flawless operation without CORS or logic errors:

### 1. Firebase Infrastructure Rules (`firestore.rules` & `cors.json`)
The Kiosk MUST be able to query the DB without user authentication, while the Backend handles all secure writing.
* **`firestore.rules` Generation:**
  ```javascript
  rules_version = '2';
  service cloud.firestore {
    match /databases/{database}/documents {
      match /printJobs/{job} {
        // Only allow backend admin SDK to write. 
        // Allow Kiosk frontend to read anonymously if querying by PIN.
        allow create, update, delete: if false; 
        allow read: if true; 
      }
    }
  }
  ```
* **`cors.json` Generation (Storage):**
  You must build a configuration file that allows `["GET"]` globally so the browser on the Lenovo Tablet can natively fetch the PDFs from Firebase Storage without CORS blocks.
  ```json
  [{"origin": ["*"], "method": ["GET"], "maxAgeSeconds": 3600}]
  ```

### 2. Backend Logic (PDF Processing, Pricing, & PIN Validation)
**IMPORTANT:** Because the backend is deployed on Vercel (Serverless Functions), it CANNOT run heavy system binaries like LibreOffice without crashing. You must structure the backend logic as follows:
1. **File Constraint:** You must enforce strict `.pdf` only uploads on the frontend dropzone AND backend validation. Reject any `.docx` dumps.
2. **Pricing Engine:** Use `pdf-lib` within the Node route to natively read the uploaded buffer and extract the total page count. Calculate the user's price mathematically using `(pages * copies * rate)`.
3. **Payment Webhook:** Upon successfully verifying the `cashfree-pg` webhook signature:
   - Generate exactly a 4-Digit PIN algorithmically: `const pin = Math.floor(1000 + Math.random() * 9000).toString();`
   - Execute a uniqueness check in Firestore to ensure the `pin` does not currently exist with a `"pending"` status.
   - Upload the PDF buffer to Firebase Storage. 
   - Push this explicit payload using `firebase-admin`:
   ```javascript
   await db.collection('printJobs').doc(pin).set({
     pin: pin,
     documentUrl: storageUrl,
     status: 'pending',
     paymentStatus: 'completed',
     createdAt: admin.firestore.FieldValue.serverTimestamp()
   });
   ```

### 3. Kiosk Hardware API Proxy (`kiosk-tablet-frontend/app/api/print/route.ts`)
The Kiosk UI cannot safely blast jobs to the Raspberry Pi over `http://` directly from an `https://` Vercel browser due to standard Mixed Content blockers. You MUST build a Next.js Server proxy:
1. The Kiosk frontend sends the matched `documentUrl` to `POST /api/print`.
2. The Vercel Server Route downloads the PDF buffer from Firebase Storage.
3. The Server Route constructs a `FormData` object containing the buffer.
4. It executes `fetch(process.env.NEXT_PUBLIC_FASTAPI_PRINT_URL, ...)` natively on the backend, thereby silently bypassing all browser CORS and Mixed Content restrictions.
